# 06 · The tool's limits

Three places where the tool can be used, but must be used carefully. Each is a real constraint, not a formality.

---

## Any interpretation is still only an interpretation

A compiled document is an interpreted version of the raw material. Every concept extracted, every edge drawn, every summary written is the result of a choice. Most of those choices are silent.

To counterweight this, every compiled sentence should keep a pointer to its source — the raw line, the date, the file. A reader should be able to doubt the machine at any moment and fall back to the original.

The bias of the interpreter is never fully removed; it is only made traceable.

---

## The atrophy of one's own noticing

If the machine finds the pattern before the person does, the person's own capacity to sit and find the pattern may be the first thing to weaken.

There is a danger in outsourcing the detection of one's own mechanicality. A system of this kind, run carelessly, can let its user stop looking — under the quiet assurance that the tool is already looking for them.

The healthy operating mode is one in which *the tool grows only as fast as the user's own noticing can keep up*.

---

## Direction does not come from compilation

None of these layers can answer the question: *what is the right question to ask?*

The machine arranges and connects; it does not choose the arrangement's purpose. Direction still rises from silence, from good teachers, from the inward encounter the tool cannot reach.

This is not a weakness of this particular tool — it is a feature of tools as such.
